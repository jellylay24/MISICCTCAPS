ICCT Cainta Campus — MIS Resource Management System

System Diagrams and Descriptions

1. System Architecture Diagram

Description: The system follows a three-tier architecture. The Presentation Tier consists of the client web browser running Tailwind CSS and Alpine.js for responsive design on desktop, tablet, and mobile devices. The Application Tier runs Laravel 11 on Apache with MVC architecture — handling routing, business logic, authentication, and server-side rendering. The Data Tier uses MySQL database for storing all system data and Brevo SMTP for email verification services. Communication between tiers uses HTTPS for security.

2. Data Flow Diagram (DFD Level 0)

Description: The DFD shows how data flows between external entities (Faculty User and Administrator) and the MIS System. The system is composed of six modules: (1) Authentication handles login and registration, (2) Resources manages resource browsing and availability display, (3) Borrow Manager processes borrowing requests with duration selection, (4) Approval enables administrators to approve/reject requests and mark items returned, (5) Notifications sends in-app alerts, and (6) Reports generates CSV exports. All modules read from and write to the MySQL Database. The Brevo Email Service is used for email verification.

3. Borrow Process Flowchart

Description: The flowchart illustrates the complete borrowing process from login to return. A faculty member logs in, browses available resources, and selects a borrowing duration (1 hour, 2 hours, 3 hours, or custom). If the resource is available, the system creates a pending request. The administrator reviews and either approves or rejects the request with a reason. Once approved, the item becomes active with a real-time countdown timer showing remaining time. The system automatically detects overdue items and displays a red alert banner. The process ends when the item is returned and marked as completed.

5. Agile SDLC Model

Description: The system followed an Agile/Iterative SDLC approach. Each phase (Planning, Analysis, Design, Development, Testing) was completed in cycles, with continuous feedback from stakeholders driving improvements. Deployment was done after each stable iteration, and maintenance continues based on user feedback.
